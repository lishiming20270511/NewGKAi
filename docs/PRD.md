## 一、产品概述

### 1.1 一句话定位

**为抖音志愿填报主播提供的一站式直播辅助工具**——主播在电脑端操作，画面以竖屏手机比例投屏到直播间，在2步填表中完成考生信息录入→AI分析→付费解锁→生成完整报告。

### 1.2 核心原则（决策引擎宪法）

```
算法负责计算（评分 + 概率）
规则引擎负责约束（过滤 + 分层）  
LLM 负责解释（不参与决策）
```

> 系统目标不是"生成推荐"，而是构建一个**基于历史数据与多因子模型的志愿决策引擎**，输出尽可能贴近真实录取结果的志愿方案。

### 1.3 用户画像（仅2类用户）

| 角色 | 身份 | 核心场景 | 使用方式 |
|------|------|----------|----------|
| **主播** (Streamer) | 抖音高考志愿填报主播 | 直播时帮考生查志愿推荐 | 电脑浏览器打开工具 → 竖屏投屏到抖音直播 |
| **管理员** (Admin) | 平台运营方 | 管理主播账号、充值、监控爬虫 | 管理后台操作 |

> **注：本产品不是 SaaS 多租户系统，是单一平台下的主播工具。一个管理员管理所有主播。**

### 1.4 核心流程

```
主播登录 → STEP1 录入考生基本信息（7个字段）
         → STEP2 填写意向偏好（专业/城市/院校/性格）
         → 点击"开始AI分析"
         → 系统前端纯JS本地生成15所推荐（不依赖后端AI）
         → 展示"付款墙"（关键信息加密）
         → 主播点击"一键解锁" → 后端扣费 → 展示完整报告
         → 可下载PDF / 测下一个考生 / 打开直播答疑
```

### 1.5 商业模式

| 项目 | 说明 |
|------|------|
| 定价 | 299元 = 10次（29.9元/次），管理员可自定义充值次数和金额 |
| 支付方式 | 主播线下转账（微信/支付宝）→ 管理员后台充值次数 |
| 全站术语 | 统一使用"**剩余次数**"（禁止使用"余额/金额"等歧义词） |
| 扣费时机 | 点击"一键解锁"时扣 1 次 |
| 在线支付 | **不做**（先走线下转账→后台充值模式） |
| 小程序 | 微信小程序"分数分析师"（洛阳强盛网络科技有限公司，已认证），待 ICP 备案通过后上线 |

### 页面②：考生信息（STEP 1/2）

**设计原则**：一屏内完成，字段精简，直播场景下1分钟填完。

| 序号 | 字段 | 组件 | 必填 | 说明 |
|:---:|------|------|:---:|------|
| 1 | 姓名/昵称 | 文本输入 | 是 | 占位提示："**考虑用户隐私，建议用网名代替**" |
| 2 | 省份 | 下拉选择（31省） | 是 | 选中后动态检测高考模式并切换选科组件 |
| 3 | 高考分数 | 数字输入 | 是 | 右侧动态显示满分（上海`/660`，其余`/750`）；自动钳制超限值 |
| 4 | 省内位次 | 数字输入 | 否 | 占位提示："可留空自动估算"；留空由系统根据一分一段表估算 |
| 5 | 选科 | 动态组件 | 是 | 省份选定后自动检测该省高考模式，动态渲染对应选科组件 |
| 6 | 服从调剂 | 双选标签 | 否 | "服从" / "不服从" 二选一 |
| 7 | 家庭经济 | 下拉选择 | 否 | 选项：较为困难 / 一般 / 良好 / 比较优越；标注"**影响院校推荐**" |

**动态选科逻辑**（依据阳光高考网 2026 年官方分类 + 对标夸克高考/掌上高考）：

| 模式 | 省数 | 全部省份 | 选科组件 |
|------|:---:|------|----------|
| **3+3** | 6 | 上海、浙江、北京、天津、**山东**、海南 | 6科标签任选3门；浙江多1门「技术」选考 |
| **3+1+2** | 25 | 河北、辽宁、江苏、福建、湖北、湖南、广东、重庆、吉林、黑龙江、安徽、江西、广西、贵州、甘肃、山西、内蒙古、**河南**、四川、云南、陕西、青海、宁夏、新疆、西藏 | 首选1门（物理/历史）+ 再选2门（化学/生物/政治/地理） |

> **⚠️ 2026年起全国31省已全部完成高考综合改革，不再存在传统"老高考/文理分科"模式。**

### 页面④：分析中

过渡动画页面——加载状态（spinner + "AI分析中…"），约1-2秒后自动跳转到付款墙。

**背后发生的事情**：
- 前端 `generateSchools()` 在浏览器本地执行推荐算法（纯JS，不依赖后端AI）
- 特别关注区构建（意向学校独立提取，不计入15所）
- 四层填充（意向城市→本省→意向城市周边→全国）
- 位次概率计算
- Tier分层（冲/稳/保）
- 15所学校排序

### 页面⑤：付款墙

**核心设计**：推荐结果已经生成，但关键内容加密遮蔽，诱使主播付费解锁。

| 区块 | 内容 |
|------|------|
| 标题 | "📋 志愿规划报告" + 报告编号 + 防伪标识 |
| 考生摘要 | 网名 · 省份 · 分数 · 选科 |
| AI洞察区 | 4个指标以"——"遮蔽：院校-分数匹配度、录取概率置信度、专业就业契合度、近6年数据覆盖率 |
| 院校列表 | "🏫 AI 推荐院校（15所）· 已加密"——学校名称可见，概率+详情全部遮蔽 |
| 剩余次数 | "剩余解锁次数：**X** 次" + "次数不足时请联系平台充值" |
| CTA按钮 | **"🔓 一键解锁完整报告"** |

**扣费流程（乐观更新）**：
```
前端 doDirectUnlock():
  ① 前端乐观扣减：localBalance -= 1，更新UI
  ② POST /auth/deduct (Bearer token)
  ③ 成功(200) → 确认，进入完整报告
  ④ 失败(非200) → 回滚 localBalance += 1，提示错误
```

## 四、决策引擎（核心算法）

### 4.1 推荐池构建

推荐结果由两个区域组成：**特别关注区**（意向学校，不计入15所） + **推荐池**（算法生成的15所）。

#### 4.1.1 特别关注区（不计入15所）

| 属性 | 说明 |
|------|------|
| 来源 | 用户填写的意向学校（最多3所） |
| 匹配方式 | 精确名称匹配（`===`），禁止模糊替代 |
| 纳入规则 | **无论概率多少一律展示**；0%概率的学校正常展示，标注"录取概率 0%" |
| 展示位置 | 报告最顶部，标记★，以独立「📌 特别关注」区域展示 |
| 与15所关系 | **不计入15所推荐总数**，推荐池独立填充15所 |

#### 4.1.2 推荐池（15所，四层填充）

| 层级 | 名称 | 来源 | 规则 |
|:---:|------|------|------|
| L1 | **意向城市学校** | 意向城市 → 城市→省份映射（60+城市映射表）→ 该省学校 | 不设概率门槛，标记●强制保留 |
| L2 | **生源地本省学校** | 考生所在省份的学校 | 去除L1已有学校，按rankProb降序取 |
| L3 | **意向城市周边** | 意向城市的地理邻省/邻市学校 | 去除L1/L2已有学校 |
| L4 | **全国兜底** | 邻省 → 沿海发达省份 → 全国 | 补足至15所 |

**优先级总原则**：**意向城市 > 本省 > 意向城市周边 > 全国**

**示例**：河南考生350分，意向城市上海，意向学校北京大学+河南科技大学+洛阳师范学院

```
特别关注区: 北京大学(0%) + 河南科技大学 + 洛阳师范学院 = 3所（不计入15所）
推荐池填充:
  L1: 上海匹配学校 = 4所 → 累计4所
  L2: 河南本省学校 = 补充8所 → 累计12所
  L3: 上海周边(江苏/浙江) = 补充3所 → 累计15所 ✅
```

### 4.2 双推荐率体系

**① 成绩排名推荐率**（`calcRankProb`，tier分层以此为准）：

```
位次比值 = 学生省内位次 / 该校历年最低位次
概率 = (1 - 位次比值) × 100 + 随机扰动(±3%)
钳制到 [1, 99]
```

**② 加权综合推荐率**（`calcWeightedProb`，六维度加权）：

```
总分 = 录取概率(35%) + 专业匹配(20%) + 就业前景(15%)
     + 城市偏好(10%) + 性格匹配(10%) + 经济适应(10%)
```

> 报告需同时展示两个推荐率，并附说明。

### 4.3 Tier 分层

| Tier | 名称 | 成绩排名推荐率范围 | 颜色 |
|:---:|------|:-----------------:|:----:|
| 0 | 🔴 冲刺 | 30% ~ 60% | 红色 |
| 1 | 🔵 稳妥 | 60% ~ 85% | 蓝色 |
| 2 | 🟢 保底 | ≥ 85% | 绿色 |

- <400分考生：阈值降至 5%
- 每tier最多5所
- 非意向学校 rankProb < 30% → 一律不推荐
- **意向学校强制展示，即使 0% 概率也保留**

### 4.4 排序优先级

1. **意向学校**（★ 特别关注区）→ 报告最顶部独立展示
2. **意向城市学校**（● `_intended_city`）→ 推荐池内置顶
3. 同 tier 内按 **rankProb** 降序排列
4. rankProb 相近（差距 ≤ 5%）时，按**性格匹配度**决胜
5. 性格未填写或匹配度相同时，按 **weightedProb** 降序

### 4.5 性格→院校类型倾向（tiebreaker 决胜规则）

| 性格标签 | 优先推荐院校类型 | 逻辑说明 |
|----------|:--------------:|----------|
| 外向活泼、社交沟通、领导管理 | **综合类/文科类** | 擅长人际交往，文科/综合类院校提供更多社交场景 |
| 沉稳内敛、逻辑分析、钻研学术 | **理工科类** | 专注力强、偏好独立钻研 |
| 艺术创作 | **艺术类/设计类** | 匹配艺术类院校 |
| 动手实践 | **工科/应用型** | 偏好实操型学习 |

### 4.6 家庭经济→院校倾向

| 家庭经济 | 专业意向条件 | 优先推荐 | 逻辑说明 |
|----------|-------------|:------:|----------|
| 较为困难 | 专业方向含师范/教育类 | **师范类院校** | 师范类有生活费补贴、免学费/低学费政策 |
| 较为困难 | 无特定师范意向 | **学费≤5000元/年公办院校** | 标注"经济友好" |
| 一般/良好/比较优越 | — | 无特殊限制 | 按正常权重评分 |

| 属性 | 值 |
|------|-----|
| 背景 | 竖向渐变紫 `#6A3FC9` → `#9368E8`，哑光无高光 |
| 衬底卡片 | 半透磨砂淡紫色，低透明度紫底 |
| 内容面板 | 纯白圆角底色 `#FFFFFF` |

| 风险 | 直播视频压缩后紫色渐变易出现色带（banding）；纯白内容面板在暗色手机屏幕上对比过强；与主流 SaaS 工具风格差异大 |

---

### 6.2 产品经理建议

**推荐方案 A（Bento Grid 暗色）作为生产版本，理由如下：**

| # | 决策依据 | 分析 |
|---|---------|------|
| 1 | **直播压缩表现** | 暗色背景经 H.264 4:2:0 采样后结构保留完整；紫色渐变在低码率下会出现明显的"色带/banding"伪影 |
| 2 | **信息可读性** | 暗底亮字天然对比度高；方案B的白色内容面板在紫色背景上"浮起"，观众容易视觉跳脱 |
| 3 | **专业信任感** | 对标产品（radar.lumenaistudio.co）为深色系；Bloomberg Terminal/Stripe Dashboard 均为暗色——家长看到暗色界面会本能认为"这是专业工具" |
| 4 | **主播体验** | 长时间直播（3-4小时），暗色界面不刺眼；方案B的紫色渐变长时间注视可能导致色彩疲劳 |

> **如果仍想尝试方案B**：建议先用 `prototype.html` 做一版紫色渐变原型，在真实直播推流环境下测试（OBS捕获 → 抖音推流 → 手机观看），对比两种方案的清晰度后再决定。

| **根因** | jsPDF的QR码生成器（`qrcode.js`）对数据有硬限制。当前QR数据 = 报告编号 `GK260615-1823`（20字符）+ 考生信息哈希（约350字符）→ 总计372字符，超过368字符上限 |
| **修复方向** | ① 缩短QR数据：仅用报告编号（~20字符），去掉哈希；② 或升级到支持更长数据的QR生成方案（QR Code Model 2 最高可容纳4296个字母数字）；③ 优先方案①，改动最小 |
| **影响范围** | 前端 `buildPDFWrap()` → QR码生成逻辑 |

#### 7.1.2 Bug #2：录取趋势超过5年限制

| 属性 | 内容 |
|------|------|
| **严重度** | 🟡 P1 — 违反PRD约束 |
| **复现** | API返回的 `admission_data.years_available` 数组包含数百个重复年份 |
| **根因** | 后端 `recommendation.py` 中 `years_available` 直接用了SQL `GROUP BY` 的原始结果，未做 `DISTINCT`。例如郑州大学的 `years_available` 中 `2025` 重复200+次、`2024` 重复100+次。`trend_detail` 将所有分数列为一长串（如 `469→533→532→575→...` 400+个值），而非精简为近5年的摘要 |
| **修复方向** | ① `years_available = sorted(set(years))[-5:]` ——去重后取最近5年；② `trend_detail` 取最近5年唯一分数值，展示格式 `2021:580→2022:596→2023:601→2024:608→2025:604（↑微升）`；③ 若不足5年，有几年来几年 |
| **影响范围** | 后端 `api/services/recommendation.py` → `_build_admission_dimensions()` |

#### 7.1.6 Bug #6：PDF报告乱码 + 内容显示不全

| 属性 | 内容 |
|------|------|
| **严重度** | 🔴 P0 — 报告不可交付 |
| **复现路径** | 任意考生→生成完整报告→点击"📥 下载PDF"→打开PDF文件 |
| **现象A：文字模糊/乱码** | 中文字体在PDF中显示为模糊笔画、缺笔画或难以辨认的像素块，英文和数字相对清晰 |
| **现象B：内容显示不全** | 报告正文第2页起的内容缺失，只有封面或第一页有内容；部分卡片右侧边缘被裁切 |

**根因分析**：

| 子问题 | 技术根因 | 详细说明 |
|--------|---------|---------|
| **文字模糊** | html2canvas `scale` 参数过低 | 当前 `scale: 1`，375px 宽的手机框截图放到 A4 纸（595pt）上被拉伸放大，导致文字像素化。需要使用 `scale: 3` 以上保证 300DPI 打印质量 |
| **中文笔画缺失** | 低分辨率下 CJK 字体渲染退化 | 中文字体笔画密度高（如"填报"的"填"有13画），在低 scale 下 html2canvas 丢失细节笔画，呈现"乱码"状 |
| **内容截断（只导出一页）** | html2canvas 仅截取视口可见区域 | 报告 HTML 渲染为长滚动页面，但 `html2canvas()` 默认只截取当前视口（viewport），超出滚动区域的学校卡片、建议书、免责声明全部丢失 |
| **右侧裁切** | 内容宽度 + 手机框 padding > 捕获区域 | 手机模拟框有左右 padding，加上某些长文本（院校全称、城市分析段落）超出375px宽度，html2canvas 截取时被裁剪 |

**修复方向**：

| 修复项 | 方案 | 优先级 |
|--------|------|:---:|
| **提升清晰度** | `html2canvas(element, { scale: 3, useCORS: true })` ——3倍缩放保证 300DPI | P0 |
| **解决内容截断** | 捕获前解除高度限制：`element.style.height = 'auto'; element.style.overflow = 'visible'`；或使用 `html2canvas` 的 `height` 参数指定完整内容高度 | P0 |
| **多页分页** | jsPDF 按 A4 高度（297mm）循环切分长图：`for(let i=0; i<totalHeight; i+=pageHeight) { pdf.addImage(crop, ...); if(i+pageHeight<totalHeight) pdf.addPage(); }` | P0 |
| **解决裁切** | 调整捕获宽度为 A4 比例（宽高比 ≈ 1:1.414），或将内容容器宽度设为 A4 物理宽度（794px@96dpi），字体等比放大 | P1 |
| **字体回退保障** | 报告中文字体统一使用系统默认中文字体（`"PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif`），避免 Web 字体加载失败导致的渲染异常 | P1 |

**影响范围**：前端 `buildPDFWrap()` → html2canvas 参数 + jsPDF 分页逻辑 + 报告容器样式
